# DreamStudio v1 产品形态梳理

本文档基于已填写的立项问卷，以及 2026-06-19 追加确认的第一阶段产品形态重新整理。

第一阶段的核心结论：

- DreamStudio 是一个多功能 AI 创作平台。
- DreamStudio 依赖用户自己的 `new-api` 密钥调用上游模型。
- DreamStudio 和 `new-api` 是两套独立系统，用户数据互不打通。
- DreamStudio v1 不处理模型成本、用户充值、支付、订阅、渠道、倍率等网关问题。
- DreamStudio v1 的重点是降低普通用户使用 `new-api` 能力的门槛，把复杂的 API 使用过程包装成易用的创作体验。

---

## 1. v1 产品定位

### 1.1 一句话定义

DreamStudio 是一个依赖用户自有 `new-api` 密钥的一站式 AI 创作平台。

### 1.2 产品解释

DreamStudio 面向已经拥有或愿意注册 `new-api` 账号的用户。用户在 `new-api` 网站完成账号注册、额度充值和密钥创建后，再注册登录 DreamStudio，并在 DreamStudio 中填写自己的 `new-api` 密钥。

之后，DreamStudio 负责提供更友好的 AI 创作界面、任务管理、资产保存和创作流程；实际模型请求由 DreamStudio 后端代理到用户配置的 `new-api` 服务。

### 1.3 DreamStudio 不是什么

- 不是大模型网关。
- 不是 `new-api` 的替代品。
- 不是 `new-api` 管理后台。
- 不是计费系统。
- v1 不是订阅制 SaaS。
- v1 不负责用户在 `new-api` 中的余额、充值、渠道和模型成本管理。

### 1.4 v1 的核心价值

- 让用户不用安装 Cherry Studio、Lobe Chat、OpenWebUI 等第三方客户端。
- 让用户不用理解复杂的客户端配置。
- 让用户通过浏览器即可进行 AI 创作。
- 让用户的创作数据、任务记录、生成结果可以在云端保存。
- 让用户以更低门槛使用自己 `new-api` 中已经接入的模型能力。

## 2. 系统边界

### 2.1 两套账号体系

v1 明确采用两套独立账号体系：

- `new-api` 账号：用于获取模型调用密钥、管理模型额度、处理充值和上游模型能力。
- DreamStudio 账号：用于登录 DreamStudio、保存创作数据、保存用户配置、管理任务和资产。

两套系统的用户不会自动绑定。DreamStudio 只保存用户主动填写的 `new-api` 连接配置。

### 2.2 用户必须准备什么

用户使用 DreamStudio 前，需要先准备：

- 一个可用的 `new-api` 账号。
- 一个有效的 `new-api` API 密钥。
- `new-api` 账号中足够的额度。
- 可访问的 `new-api` 服务地址。

### 2.3 DreamStudio 负责什么

DreamStudio v1 负责：

- 用户注册与登录。
- 保存用户的 `new-api` 连接配置。
- 提供 AI 绘画创作界面。
- 提供任务提交、任务状态、失败提示和重试能力。
- 保存用户上传的参考图。
- 保存生成结果图。
- 提供资产仓库。
- 提供基础个人中心。
- 提供管理员后台。
- 提供系统级存储、任务超时、重试次数等配置。
- 提供请求日志，方便管理员定位问题。

### 2.4 new-api 负责什么

`new-api` 负责：

- 模型厂商接入。
- 模型统一接口。
- 用户在 `new-api` 侧的额度。
- 用户在 `new-api` 侧的充值。
- 用户在 `new-api` 侧的 API 密钥。
- 上游模型调用与渠道路由。
- 上游模型成本与调用失败。

### 2.5 DreamStudio v1 不负责什么

DreamStudio v1 不负责：

- 给用户充值。
- 给用户售卖额度。
- 计算模型成本。
- 做模型渠道管理。
- 做模型倍率配置。
- 代理 `new-api` 的完整后台能力。
- 替用户创建 `new-api` 账号。
- 替用户创建 `new-api` 密钥。
- 管理用户在 `new-api` 中的余额。

## 3. v1 用户流程

### 3.1 新用户首次使用流程

1. 用户访问 DreamStudio 首页。
2. 用户点击开始使用。
3. 系统跳转到登录/注册页。
4. 用户使用账号密码注册 DreamStudio。
5. 用户登录 DreamStudio。
6. 系统检测用户未配置 `new-api` 密钥。
7. 系统弹窗引导用户填写 `new-api` 服务地址和 API 密钥。
8. 用户点击引导链接前往 `new-api` 网站。
9. 用户在 `new-api` 中注册或登录，创建 API 密钥。
10. 用户回到 DreamStudio，填写服务地址和密钥。
11. DreamStudio 测试连接和密钥是否可用。
12. 测试通过后，用户进入创作台。

### 3.2 再次使用流程

1. 用户登录 DreamStudio。
2. 系统读取用户已保存的 `new-api` 配置。
3. 用户进入 AI 创作台。
4. 用户选择模型和创作参数。
5. 用户提交生成任务。
6. DreamStudio 后端使用用户密钥向 `new-api` 发起请求。
7. 任务完成后，DreamStudio 保存结果并展示给用户。

### 3.3 AI 绘画核心流程

1. 用户进入 AI 创作台。
2. 用户选择图片模型。
3. 用户设置生成参数，例如尺寸、比例、分辨率、生成张数。
4. 用户可选上传参考图。
5. 用户输入提示词。
6. 用户点击生成。
7. DreamStudio 创建异步生成任务。
8. DreamStudio 后端调用用户配置的 `new-api`。
9. 系统显示任务处理中状态。
10. 任务成功后展示结果图。
11. DreamStudio 将结果图保存到资产仓库。
12. 用户可以查看、下载、批量删除生成结果。

### 3.4 额度不足流程

1. DreamStudio 调用 `new-api` 失败。
2. 系统识别错误可能与额度不足有关。
3. 页面提示用户：`new-api` 额度不足，请前往 `new-api` 检查余额或充值。
4. 提供跳转到用户配置的 `new-api` 服务地址的入口。
5. DreamStudio 不处理充值和支付。

### 3.5 密钥失效流程

1. DreamStudio 调用 `new-api` 失败。
2. 系统识别错误可能与密钥失效、密钥错误或服务地址不可用有关。
3. 页面提示用户检查 `new-api` 服务地址和 API 密钥。
4. 用户进入个人中心重新配置密钥。
5. 系统重新测试连接。

- AI 绘画创作台。
- 图片模型选择。
- 绘画参数设置。
- 参考图上传。
- 提示词输入。
- 异步生图任务。

v1 可以在数据结构或页面入口上预留：

- 后续订阅。
- 后续会员等级。
- 后续视频生成。
- 后续 AI 对话。

预留的原则：

- 可以在数据库设计中保留扩展字段。
- 可以在代码结构中留出模块边界。
- 不在 v1 页面中暴露未完成能力。
- 不为了未来功能拖慢 v1 主流程。

## 5. 页面结构

### 5.1 页面清单

v1 页面建议：

- 首页。
- 登录页。
- 注册页。
- AI 创作台。
- 任务页。
- 资产仓库。
- 个人中心。
- new-api 配置页。
- 管理后台。
- 管理后台用户管理页。
- 管理后台系统设置页。
- 管理后台存储设置页。
- 管理后台请求日志页。

### 5.2 首页

首页定位：官网型入口。

首页目标：

- 让用户理解 DreamStudio 是一个 AI 创作平台。
- 让用户知道使用前需要准备 `new-api` 密钥。
- 引导用户开始使用。

首页不需要直接提供可用的生成入口。

### 5.3 AI 创作台

AI 创作台是 v1 的核心页面。

建议结构：

- 左侧：模型分类和模型列表。
- 中间：创作输入区、参考图、参数区。
- 右侧或下方：任务状态与生成结果。
- 顶部：当前用户、密钥状态、进入个人中心入口。

v1 左侧分类建议先保留：

- 图片。

后续再扩展：

- 聊天。
- 视频。

### 5.4 资产仓库

资产仓库 v1 负责展示用户生成过的图片和上传过的参考图。

v1 支持：

- 查看图片。
- 下载图片。
- 批量删除。
- 按任务状态筛选：成功、失败、处理中。

v1 暂不支持：

- 文件夹。
- 标签。
- 搜索。
- 分享链接。
- 回收站。

### 5.5 个人中心

个人中心 v1 支持：

- 查看基本账号信息。
- 修改密码。
- 配置 `new-api` 服务地址。
- 配置 `new-api` API 密钥。
- 测试 `new-api` 连接。
- 查看当前密钥状态。

### 5.6 管理后台

管理后台 v1 支持：

- 用户列表。
- 启用用户。
- 禁用用户。
- 查看请求日志。
- 配置本地存储。
- 配置 S3 存储。
- 配置参考图保存策略。
- 配置结果图保存策略。
- 配置生图超时时间。
- 配置生图重试次数。

## 6. 账号与权限

### 6.1 DreamStudio 账号

v1 支持：

- 用户名密码注册。
- 用户名密码登录。
- 普通用户。
- 超级管理员。
- 用户启用/禁用。

v1 不支持：

- 邮箱验证。
- 手机验证码。
- OAuth 登录。
- 2FA。
- 登录设备管理。
- 异地登录提醒。
- 用户自行注销账号。
- 用户自行删除全部数据。

### 6.2 角色

v1 建议只保留两个角色：

- 普通用户：使用 AI 创作台、任务页、资产仓库、个人中心。
- 超级管理员：拥有普通用户能力，并可进入管理后台。

问卷中提到的“会员”建议 v1 暂不落地，留给后续订阅版本。

### 6.3 密钥权限

用户的 `new-api` API 密钥只归该用户使用。

DreamStudio 后端在执行模型请求时，需要根据当前登录用户读取对应的密钥配置，不能使用其他用户的密钥，也不能使用系统统一密钥替代用户密钥。

## 7. new-api 配置设计

### 7.1 用户侧配置项

每个 DreamStudio 用户需要配置：

- `new-api` 服务地址。
- `new-api` API 密钥。
- 是否启用该配置。
- 最近一次连接测试状态。
- 最近一次连接测试时间。

### 7.2 连接测试

用户保存配置时，系统应执行连接测试。

测试目标：

- 服务地址可访问。
- API 密钥有效。
- 能通过 `/v1/models` 获取模型列表，避免连接测试消耗额度。

测试结果：

- 成功：允许进入创作台。
- 失败：保存配置但标记为不可用，提示用户修正。

### 7.3 密钥保存

v1 必须将 API 密钥作为敏感数据处理。

建议：

- 数据库存储加密后的密钥。
- 页面展示时只显示掩码。
- 日志中禁止打印完整密钥。
- 请求失败日志中禁止包含 Authorization 明文。
- 管理员可代用户配置或重置 `new-api` 密钥，但不能查看已保存密钥明文。
- 管理员代配置或重置密钥时必须记录审计日志。

### 7.4 服务地址

用户需要填写自己的 `new-api` 服务地址。

原因：

- 不同用户可能使用不同 `new-api` 站点。
- DreamStudio v1 不绑定单一 `new-api` 实例。
- 这能让 DreamStudio 更像通用创作平台，而不是某个 `new-api` 站点的附属前台。

已确认：

- 管理员必须配置默认 `new-api` 服务地址。
- 默认地址不写死在代码或产品文档中，通过环境变量或管理员初始化配置提供。
- 是否允许用户自定义 `new-api` 服务地址由管理员开关控制。
- 管理员关闭用户自定义服务地址时，用户只填写 API 密钥，并使用默认服务地址。

## 8. 数据与存储

### 8.1 DreamStudio 需要保存的数据

v1 保存：

- 用户账号。
- 用户角色。
- 用户启用状态。
- 用户 `new-api` 配置。
- AI 绘画任务。
- 任务请求参数。
- 任务状态。
- 错误摘要。
- 上传参考图。
- 生成结果图。
- 资产记录。
- 请求日志。
- 系统设置。
- 存储设置。

### 8.2 数据库

问卷中倾向使用 PostgreSQL 17。

v1 建议：

- 使用 PostgreSQL 作为主数据库。
- 使用 Redis 处理任务状态缓存、队列或锁。
- 使用对象存储或本地存储保存图片文件。

### 8.3 存储方式

v1 需要支持：

- 本地存储。
- S3 兼容对象存储。

需要区分：

- 参考图存储。
- 结果图存储。

管理员可配置：

- 存储方式。
- 本地存储路径。
- S3 endpoint。
- S3 bucket。
- S3 region。
- S3 access key。
- S3 secret key。
- S3 public base URL。
- 文件保留时间，单位为小时。

### 8.4 保留策略

v1 默认：

- 会话和任务记录长期保存。
- 上传文件和生成结果可由管理员设置保留时间。
- 删除后不可恢复。
- 不提供回收站。

后续订阅版本可扩展：

- 会员有效期内长期保存。
- 会员过期后进入缓冲保留期。
- 缓冲期后自动清理。

## 9. 任务与日志

### 9.1 异步任务

AI 绘画应作为异步任务处理。

原因：

- 生图耗时不可控。
- 用户网络中断不应导致任务丢失。
- 结果需要保存到资产仓库。
- 失败和重试需要可追踪。

### 9.2 任务状态

建议 v1 任务状态：

- `pending`：等待处理。
- `running`：处理中。
- `succeeded`：成功。
- `failed`：失败。
- `timeout`：超时。
- `canceled`：取消。

### 9.3 重试策略

管理员可配置：

- 单次任务超时时间。
- 最大重试次数。
- 重试间隔。

任务失败时应记录：

- 失败阶段。
- 错误摘要。
- 上游响应状态码。
- 是否可重试。

### 9.4 请求日志

v1 请求日志主要给管理员定位故障。

日志建议记录：

- 请求用户。
- 请求类型。
- 模型名称。
- 请求时间。
- 响应时间。
- 任务 ID。
- 状态。
- 错误摘要。
- `new-api` 服务地址域名。

日志禁止记录：

- 完整 API 密钥。
- 完整 Authorization Header。
- 不必要的敏感请求体。

## 10. 技术方向

### 10.1 前端

问卷中倾向：

- Next.js。
- React。
- TypeScript。
- Tailwind CSS。
- shadcn/ui。

v1 前端重点：

- 桌面 Web 优先。
- 移动端 Web 适配。
- 暗色视觉风格。
- 创作感和工具感。
- 默认隐藏高级参数。

### 10.2 后端

v1 推荐采用独立后端服务，而不是只依赖 Next.js Route Handlers 承担全部后端职责。

推荐方向：

- NestJS。
- TypeScript。
- Prisma。
- PostgreSQL 17。
- Redis。
- BullMQ。

推荐原因：

- DreamStudio v1 的核心不是普通 CRUD，而是异步生图任务、重试、超时、文件上传、对象存储、日志和后台管理。
- NestJS 更适合承载较完整的业务后端、模块化权限和任务服务。
- BullMQ + Redis 更适合处理生图这种耗时任务，用户关闭页面后任务仍可继续执行。
- Prisma 可以统一管理 PostgreSQL 数据模型和迁移。
- 前后端都使用 TypeScript，有利于早期降低开发成本。

v1 后端必须支持：

- 用户认证。
- 权限控制。
- 用户密钥加密保存。
- 代理请求 `new-api`。
- 异步任务。
- 文件上传。
- 图片存储。
- 请求日志。
- 管理后台 API。

Next.js Route Handlers 可以用于：

- 轻量页面辅助接口。
- BFF 聚合接口。
- 鉴权状态读取。
- 前端同源代理。

Next.js Route Handlers 不建议用于：

- 长时间运行的生图任务。
- 队列消费者。
- 文件清理任务。
- 定时任务。
- 复杂后台管理业务。

### 10.3 部署

v1 必须支持：

- Docker Compose 部署。
- DreamStudio 和 `new-api` 分开部署。
- PostgreSQL。
- Redis。
- 本地存储或 S3 存储。

v1 暂不要求：

- CI/CD。
- 自动回滚。
- 多环境。
- 监控告警。

但建议保留：

- 错误追踪。
- 基础健康检查。
- 容器日志。

### 10.4 推荐部署形态

v1 建议采用一个 DreamStudio 应用容器，对外作为一个部署单元。容器内部包含三类逻辑模块：

- `web module`：Next.js 前端，负责用户界面、管理后台页面和轻量 BFF。
- `api module`：NestJS 后端，负责认证、业务 API、任务创建、日志、资产、系统设置。
- `worker module`：NestJS Worker，负责异步生图、重试、超时、结果保存、文件清理。

基础依赖：

- PostgreSQL：主数据库，可以使用 Docker Compose 本地 PostgreSQL，也可以使用云 PostgreSQL。
- Redis：队列、任务状态和缓存，可以使用 Docker Compose 本地 Redis，也可以使用云 Redis。
- `storage`：本地挂载目录或 S3 兼容对象存储。

Docker Compose 首版应至少能启动：

- `dreamstudio` 单应用容器。

如果使用本地依赖，Compose 也可以同时启动：

- PostgreSQL。
- Redis。

如果使用云数据库和云 Redis，Compose 只需要启动 DreamStudio 应用容器，并通过 `DATABASE_URL`、`REDIS_URL` 连接外部服务。

## 11. 设计风格

### 11.1 参考方向

参考产品：

- 即梦网页版。

期望风格：

- 暗色。
- 极简科技。
- 高级感。

不喜欢：

- 配置感太强的技术后台。
- 广告和推广干扰。

### 11.2 v1 设计原则

- 创作入口要直接。
- 高级参数默认隐藏。
- 不让用户看到过多技术概念。
- 密钥配置只在首次引导和个人中心出现。
- 管理后台与用户创作台视觉上分离。

## 12. 当前冲突项与调整结论

### 12.1 关于“用户是否知道 new-api”

原问卷中写到“终端用户完全不知道底层是 new-api”。

但新的 v1 产品形态要求用户必须先注册 `new-api` 并填写 `new-api` 密钥。

因此 v1 调整为：

- 用户必须知道自己使用的是 `new-api` 密钥。
- 普通创作流程中不反复强调 `new-api`。
- 密钥配置、额度不足、密钥失效等场景必须明确提示 `new-api`。
- 对外营销可弱化技术概念，但不能隐藏必要依赖。

### 12.2 关于“计费”

原问卷中提到未来可能做充值 + 订阅混合。

新的 v1 产品形态调整为：

- v1 不做 DreamStudio 侧支付。
- v1 不做 DreamStudio 侧订阅。
- v1 不做订单。
- v1 不计算模型成本。
- 用户只需要保证自己的 `new-api` 额度充足。

后续版本再考虑：

- DreamStudio 订阅。
- 存储会员。
- 高级功能会员。
- 使用 DreamStudio 自有统一密钥的托管模式。

### 12.3 关于“首版功能”

原问卷中聊天、视频、多轮对话等能力有部分回答为“支持”。

新的 v1 产品形态调整为：

- v1 只聚焦 AI 绘画。
- 视频生成放到 v2。
- AI 对话放到 v2 或 v3。
- 多轮对话、流式输出、Markdown、代码块等能力暂不进入 v1。

### 12.4 关于“角色”

原问卷中包含游客、普通用户、会员、超级管理员。

v1 调整为：

- 普通用户。
- 超级管理员。

游客和会员暂不实现。

## 13. 待继续确认的问题

这些问题会影响下一版 PRD 和架构设计。

### 13.1 new-api 服务地址

- DreamStudio 是否允许每个用户填写不同的 `new-api` 服务地址？ 允许，但是需要由管理员控制，管理员可以设置默认API地址，是否允许用户使用自定义api地址，如果未开启，则用户端不显示对应的自定义api地址参数设置，只能设置自定义密钥并使用默认的api地址
- 是否需要管理员配置一个默认 `new-api` 服务地址？  是
- 如果管理员配置默认地址，用户是否可以覆盖？是，但是需要管理员选择是否允许用户设置

### 13.2 模型列表

- 图片模型列表从哪里来？ 通过new-api的接口获得，然后选择制定的模型
- 是从用户的 `new-api` 动态获取，还是由 DreamStudio 管理员手动配置？管理员手动配置，但是可以通过new-api的接口统一获取
- 如果从 `new-api` 获取，如何判断哪些模型是图片模型？管理员配置
- 是否需要管理员维护“模型分类”和“推荐模型”？是

### 13.3 生图接口

- v1 主要对接 `new-api` 的哪个图片生成接口？第一验收路径为 `/v1/images/generations` 和 `/v1/images/edits`；`/v1beta/models/{model}:generateContent` 放入 v1 后段兼容目标。
- 是否需要兼容多个图片生成接口格式？  是
- 参考图上传后，是由 DreamStudio 上传给 `new-api`，还是先保存到 DreamStudio 再传 URL？由 DreamStudio 上传给 `new-api`

### 13.4 任务模式

- `new-api` 的图片生成请求是同步返回，还是可能异步返回？
- DreamStudio 是否统一包装成异步任务，不管上游同步还是异步？统一包装为异步
- 用户关闭页面后，是否需要任务继续执行？继续

### 13.5 密钥安全

- 是否允许超级管理员查看用户配置的 `new-api` 密钥明文？否。管理员可代用户配置或重置密钥，但不能查看已保存密钥明文；相关操作需要记录审计日志。
- 用户密钥加密的主密钥如何配置？待定
- 如果主密钥丢失，是否接受用户重新填写密钥？是

### 13.6 首页表达

- 首页是否直接告诉用户“需要准备 new-api 密钥”？否
- 还是在点击开始使用后再提示？点击开始后
- 首页第一屏更强调“一站式 AI 创作平台”，还是强调“连接你的 new-api，开始创作”？一站式 AI 创作平台

### 13.7 后端技术栈

- 后端是否倾向使用 Go？否。v1 暂不优先使用 Go，避免前后端技术栈分裂过早增加成本。
- 是否倾向使用 Node.js / NestJS？是。推荐 v1 使用 NestJS 作为独立后端服务。
- 是否倾向使用 Next.js API Routes 先做一体化 MVP？不作为核心后端。Next.js Route Handlers 可用于轻量 BFF，但不承担异步任务、队列消费者和后台业务主逻辑。
- 是否需要从一开始就按前后端独立服务设计？逻辑上分为 Next.js Web、NestJS API、NestJS Worker 三个模块；部署上 v1 采用一个 `dreamstudio` 单应用容器。
